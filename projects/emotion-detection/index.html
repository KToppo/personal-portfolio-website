Emotion Detection Without Deep Learning

A deliberate constraint project.

Most emotion detection systems today rely on deep learning by default. I deliberately chose not to.

This project started as a final assessment during my Data Science program at IT-Vedant, but very quickly turned into a personal challenge: Can a facial emotion detection system be built using only classical machine learning — and still work in real-world conditions?

Instead of taking the obvious CNN route, I restricted myself to traditional ML models to deeply understand feature extraction, data imbalance, model behavior, and ensemble decision-making.

The Core Problem (and Why It Was Non-Trivial)

Real-time facial emotion detection using classical machine learning and ensemble decision-making.

Emotion datasets are inherently imbalanced. Some emotions dominate, others barely exist. Training a classifier naively gives high overall accuracy but completely fails minority emotions — which defeats the entire purpose.

I experimented with:

  • SMOTE — Balanced class distribution, but weak real-world performance.
  • SMOTEENN — Better overall scores, but unstable class-wise predictions.

Neither approach alone was acceptable. That’s when I stopped chasing a single “best model” and shifted to model behavior analysis.

My Solution: Stacking Based on Failure Patterns

Instead of forcing one algorithm to do everything, I trained five different classical ML models using:

  • Different resampling strategies
  • Balanced vs weighted class_weight configurations
  • Models with distinct strengths across emotion classes

Each model failed differently — and that was the insight.

I stacked these models using:

  • Majority voting
  • Probability aggregation

This ensemble approach significantly improved robustness across emotions, even when individual models struggled. This was not accidental performance — it was engineered diversity.

Real-World Accuracy: Closing the Face Detection Gap

Another major issue surfaced during testing. The training dataset contained already cropped faces — real images do not.

To close this gap:

  • Used the same Haar Cascade face detection model used by the dataset creator
  • Implemented face detection and cropping during inference
  • Ensured consistency between training and prediction pipelines

This significantly improved real-world usability. That said, I documented limitations intentionally:

  • Extremely close or poorly framed faces can fail detection
  • Some misclassification occurs due to bounding box distortion

Hiding these would have been dishonest.

What This Project Actually Demonstrates

  • Practical handling of imbalanced classification problems
  • When and why resampling techniques fail
  • How ensemble learning compensates for individual model weaknesses
  • The importance of data pipeline consistency
  • Trade-offs between accuracy, interpretability, and robustness

Most importantly, it demonstrates decision-making under constraints, not blind algorithm usage.

Why I’m Proud of This Project

Not because it’s perfect — it isn’t. But because every design choice was intentional, tested, and justified.

I don’t blindly follow trends. I analyze failure before optimizing success. I care about real-world behavior, not just leaderboard scores.
